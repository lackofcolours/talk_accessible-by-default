What is
web accessibility?

Inclusion and access to
information and services

I love digital.govt.nz‘s explanation:

Web accessibility is about inclusion — making sure everyone, including disabled people and those using assistive technologies, can access online information and services.

So what this is saying, people need to be able to access and use the web.

Why is it important that everyone have access? Because

Information leads to knowledge. Knowledge empowers to people

What about services? Services enable users to achieve their goal

How is
web = accessible?

Hyper Text Markup Language
🤍 HTML

How were these information and services made accessible?

By using declarative programming languages like Hyper Text Markup Language (HTML)

HTML is a way to give meaning to text content!

Why is HTML accessible?

Being a declarative language by nature, it was designed to be mostly backward compatible, and simpler to use for developers/content creators.

It’s also standardised so any browser can interpret and render it...

Yes, even your smart fridge’s browser.

It might not know how to render JavaScript or even Cascading Style Sheets, but it sure knows how to render HTML.

Who is web accessibility for?

🤍 Everyone

Who’s all this work for?

If you’ve been listening, or have used the internet lately, that’s technically everyone.

It IS the purpose of the web. Well coded products with HTML, CSS, and JS enable users to access web content.

If the web was coded in a non-standard way, our laptops, phone, and smart toaster browser will not be able to render the content for us to consume well.

It slightly disables or hinders our ability to use the web.

But meaningless HTML, hiding content via CSS, and blocking JS can be especially disabling for the disabled communities.

“Disabled”
communities

Permanent, temporary, situational

When I started learning about disability, I was so scared of referring to someone as “disabled”.

Is that an offensive way to refer to a group?

But after talking to advocates, reading user stories, and watching demos, I realised no, “disabled people” is exactly what we should be using.

It’s social model of disability, which is especially relevant when us creators create things for people to use.

  • A blind person can be disabled from using a site that isn’t well designed for screen readers.
  • A wheel chair rider post hip surgery can be disabled by narrow groceries self checkout.
  • A person cooking with dirty hands could be disabled from using recipe software if it doesn’t allow for voice dictation.
  • A person with severe ADHD was disabled from finding product information, because of heavy use of distracting looping animations in the marketing site.

In all of these usage of the term “disabled”, you’ll notice that it’s how the technologies have been designed that disbled and barred users with different access needs from helping themselves.

By not considering, or designing with these users, design stops these communities from becoming independent.

It is the creators saying: The problem is with you, not the tool.

Wow, what a way to create an inclusive world.

It’s especially scary to think about, when anyone can become ‘disabled’ at any point in their lives.

🪄 Assistive tech

So, how do these people with access needs consume the web?

They use assistive technologies!

There are many many types of assistive technologies to serve people with different access needs.

A disabled user may adopt different strategies to access content, which could use a variety of tools.

Example of the most common tools out there:

1. Glasses

Oh yes, what I’m wearing now is an assistive tech.

In fact, one of the most common and oldest forms still in use today!

I wear it so that I can read content without the aid of other people.

Useful for working out if I need to say hi to someone walking towards me.

Now, lets move a little closer to other tech which helps users in the digital era.

2. Screen Magnification

No, not just Control + to zoom into different apps or documents.

But the whole computer could be zoomed in.

Users with low or tunnel vision, need to be able to consistently view the whole computer in predictable ways. Page reflows when you just zoom into specific app don't really help with that.

And marvel at how some multi column apps just aren’t usable this way.

Remember that content writing guideline of no more than 8-12 sentences in one line?

Yeah, these users especially need that to happen.

And single column layout with all important functions on the left for L-to-R language speakers.

Wow single column layout has never been so important.

3. High contrast mode

What it says on the tin.

This is how the users who struggle with shades of grey designs use to cope.

Since they’ve felt they can’t rely on the site creators to provide an accessible colour palette for them.

This means the colours could be inverted automatically. Images without contrast alternatives could blur into the background.

4. Refreshable Braille display

Watching someone use one of these was so magical.

A deaf and blind user can plug these braille displays into their computer or phone, and it translate what screen readers read out into braille!

A lot of these readers have amazing shortcuts. Advance users can type whole paragraphs purely with a couple of keystrokes!!!

Since we know almost definitely that braille users can’t see, it’s more important than ever to make sure everything has a text description, and is in logical order, so that it is easy to understand when read in order.

Last - Voice Dictation

You know when you tell Alexa or Siri to do stuff?

That’s a kind of voice dictation tool.

There are others, like Dragon Naturally Speaking and Mac Voice Command,

which let users who can’t or don't want to type,

interact with the whole OS by voice.

Imagine going on a shopping site and doing hands free shopping by saying ‘Click buy banana?

In this screenshot, I can say number of the grid on the screen, to zoom into, or select that area

How do we definitely know if our product is accessible?

There are a few ways. We could...

Ask users

Yeah sure… one concrete way to find out is mass usability testing with real users.

They tell you if your stuff is not usable!

But! Usability testing at this point is a waste of time and money. Like all good research, you need to narrow down your problems

Normal controlled sample size usability testing session is already a labour intensive process.

Testing now, is like asking a user, is my app usable? There are more affordable ways

That’s where Web Content Accessibility Guidelines come in.

lots of fragment

  • Roster devs
  • Turn them into observers and/or note taker
