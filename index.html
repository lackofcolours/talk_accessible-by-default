How do we definitely know if our product is accessible?

There are a few ways. We could...

Ask users

Yeah sure‚Ä¶ one concrete way to find out is mass usability testing with real users.

They tell you if your stuff is not usable!

But! Usability testing at this point is a waste of time and money. Like all good research, you need to narrow down your problems

Testing now, is like asking a user, is my app usable? There are more affordable ways

That‚Äôs where Web Content Accessibility Guidelines come in.

WCAG
POUR principles

Perceivable, Operable, Understandable, and Robust

Web Content Accessibility Guidelines (WCAG)

The guideline is true to its 4 core principles. POUR

  • Perceivable information and user interface, text and text alternative content, etc.
  • Operable user interface and navigation, can be easily navigated using multiple types of input device. Mouse, Keyboard, voice, etc.
  • Understandable information and user interface, clear and predictable
  • Robust content and reliable interpretation by assistive tech and future tools

These principles translate into almost 50 individual success criteria - checklists by W3C that measure how accessible your content is.

The guidelines also come in different versions, like WCAG 2.0, 2.1, 2.2 and the upcoming 3.0. You can also meet different levels of conformance in the guide, like A (minimal), AA (acceptable), and AAA (Optimal experience).

WCAG success criteria and the fix is not a 1-1 relationship.

That‚Äôs a lot of redundant and duplicate admin work to create these types of error based tasks.

3. Requirements -
User story

My personal recommendation is to write them as user stories.

When writing a story, look at the failed criteria, and visualise how these criteria help users with different types of access needs. Useful questions to ask:

  • Which type of user are we disabling, if this failed WCAG/issue is not fixed?
  • What kind of strategy/tools would these users use to consume content?
  • What do we need to do to serve those users the best possible user experience?

    • Who could we be excluding?

Based on those answers, write tasks up as user stories.

This will give the team more information and guidance when looking for an appropriate solution and narrow down testing methods.

This makes it much easier for the team to deal with issues, and chip away at the errors.
